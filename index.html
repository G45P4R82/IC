Metodologia de Mineração (MSR)

Para garantir a reprodutibilidade do estudo, a extração de dados foi realizada utilizando a GitHub CLI (gh) e a GitHub REST API, sem a necessidade de clonar os repositórios localmente.

1. Estratégia de Busca e Seleção

As seguintes queries foram executadas para garantir diversidade arquitetural de repositórios modernos (criados em 2025 e 2026):

Modelos Embarcados (Níveis 3 e 4)

gh api "/search/repositories?q=fastapi+machine-learning+created:>2025-01-01&sort=stars&per_page=10"

Agnósticos (ONNX - Nível 2)

gh api "/search/repositories?q=fastapi+onnx+created:>2025-01-01&sort=stars&per_page=10"

Servidor de Inferência e Cloud (Nível 1)

gh api "/search/repositories?q=fastapi+mlflow+created:>2025-01-01&sort=stars&per_page=10"
gh api "/search/repositories?q=fastapi+boto3+created:>2025-01-01&sort=stars&per_page=10"

2. Extração de Métricas (AST Simulado)

O script realiza as seguintes extrações de evidências nos arquivos Python:

  • Fan-out (Imports Totais): Contagem de import X e from X import Y.
  • Imports de ML: Filtro específico para bibliotecas (sklearn, pandas, numpy, joblib).
  • Chamadas Diretas de ML: Busca por invocações como .predict() e joblib.load().
  • Dependência de Serviço: Busca por clientes HTTP (requests.get, boto3.invoke_endpoint).

Conclusões Arquiteturais

1. Predominância do Acoplamento Forte: Na maioria das APIs de ML (Nível 4), a aplicação importa bibliotecas matemáticas pesadas diretamente na rota.

2. Ferramentas MLOps vs. Isolamento: A presença de ferramentas como MLflow ou Docker não garante o isolamento do modelo no código-fonte Python.

3. Padrões de Baixo Acoplamento: Alcançados ao exportar para grafos padronizados (ONNX) ou ao delegar a inferência para microsserviços/Cloud (AWS Boto3, Triton).
